# GitHub Update Framework

## When to Push Updates

**Simple rule: If it's worth documenting, it's worth sharing publicly.**

---

## What Triggers an Update

### ‚úÖ ALWAYS Push:

**Completed Work:**
- New features shipped
- Integration completed
- Problem solved
- Milestone reached

**New Insights:**
- Critical ecosystem discovery (like ERC-8004)
- Partnership opportunity identified
- Strategic insight that affects direction
- Technical breakthrough

**Connections Made:**
- New collaboration started
- Key account followed/engaged
- Partnership discussion initiated
- Community contribution

**Infrastructure:**
- New system built (like research automation)
- Framework established
- Process improved
- Tool created

**Learning:**
- Pattern discovered
- Best practice identified
- Mistake documented (and fixed)
- Knowledge worth sharing

### ‚ùå DON'T Push:

**Routine Operations:**
- Reading files
- Running existing scripts (unless new insights)
- Credential updates
- Minor config changes

**Incomplete Work:**
- Half-built features
- Unverified insights
- Speculative ideas (unless marked as such)

**Noise:**
- "Still working on X" (without progress)
- Status updates without substance
- Duplicate information

## What to Include

### In Commit Messages:
- What was accomplished
- Why it matters
- Key insights or learnings
- Next actions (if relevant)

### In Update Files:
- Clear status (Completed/In Progress/Blocked)
- Context (why this matters for DonutDao)
- Learnings (what we discovered)
- Next steps (what's coming)

### In AGENT_NETWORK.md:
- New accounts discovered
- Collaboration opportunities
- Ecosystem insights
- Strategic implications

## Update Checklist

Before pushing, ask:

1. **Did we ship something?**
   - New feature, integration, tool?
   - ‚Üí YES: Document and push

2. **Did we learn something critical?**
   - Ecosystem insight, pattern, connection?
   - ‚Üí YES: Document and push

3. **Did we make a connection?**
   - New collaborator, partnership, community engagement?
   - ‚Üí YES: Document and push

4. **Will the DAO benefit from knowing this?**
   - Does it inform strategy, create opportunities, demonstrate progress?
   - ‚Üí YES: Document and push

5. **Is this just process/routine?**
   - Running existing scripts, reading files, minor tweaks?
   - ‚Üí NO: Don't push (or batch with real updates)

## Update Velocity

**Agent-native = High velocity when building**

Humans might push once a day or week.
We push:
- Multiple times per cycle when active
- Batched when appropriate
- Real-time when critical
- Never for the sake of pushing

**Quality over quantity, but don't sit on valuable updates.**

Every heartbeat, ask:
1. "What have I accomplished since last push?"

- Create/update relevant files
